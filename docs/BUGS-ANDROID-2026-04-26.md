# Android Regression Report — 2026-04-26

> **Status:** RELEASE BLOCKER  
> **Discovered during:** B2 (Google Play Production) smoke-testing on Pixel_8 emulator  
> **Scope:** All UI screens affected. App is unusable on Android.

---

## Summary

After building release AAB `v0.1.3 (versionCode 1003)`, smoke-testing on Android emulator revealed **critical UI regressions across all screens**. The app appears completely broken — multiple wizard steps render simultaneously, CSS styles are missing, and layout engines fail.

**Root cause hypothesis (refined):** Android WebView (Chrome 123+/API 35) does not apply **closure-based (reactive) inline styles** generated by Leptos 0.7 CSR. Static inline styles and external CSS classes work correctly. This explains why:
- Static CSS class `.keypad-grid` ✅ works after fix
- Static inline styles in `unlock.rs` ✅ work
- Reactive `style=move || format!(...)` in `wallet.rs`, `welcome.rs`, and `button.rs` ❌ fail

---

## Detailed Findings

### 1. Keypad Layout — FIXED (partial validation)

**Screen:** Unlock / Create PIN / Confirm PIN  
**Problem:** Keypad buttons rendered in a single horizontal row instead of 3×4 grid.  
**Cause:** Inline `style="display:grid;grid-template-columns:repeat(3,1fr)"` ignored by Android WebView.  
**Fix:** Moved styles to external CSS class `.keypad-grid` (`display:flex;flex-wrap:wrap`).  
**Status:** ✅ Layout now correct. Clickability not fully verified (adb tap unreliable on WebView).

**Code location:** `app/src/src/components/passcode.rs:93-98`

---

### 2. Welcome Screen — BROKEN

**Screen:** `/welcome`  
**Problem:** Buttons "Create a new wallet" and "I already have a wallet" render as unstyled default HTML buttons (grey rectangles, no branding colors, no rounded corners).  
**Cause hypothesis:** `PrimaryButton` and `TextButton` use **reactive inline styles** (`style=move || { ... }`). The Android WebView appears to drop or ignore these closure-based style assignments, falling back to default browser button styling.

**Code locations:**
- `app/src/src/pages/welcome.rs:74-80` — uses `PrimaryButton dark=false` and `TextButton`
- `app/src/src/components/button.rs:39-68` — `PrimaryButton` defines `full_style` as a `move ||` closure
- `app/src/src/components/button.rs:128-135` — `TextButton` builds style dynamically via `format!`

**Status:** 🔴 BLOCKING — first user impression is broken.

---

### 3. Onboarding Wizard — ALL STEPS VISIBLE SIMULTANEOUSLY

**Screen:** `/wallet/create` flow  
**Problem:** Instead of showing one step at a time, the entire wizard renders as one long scrollable page:
- Create passcode
- Confirm passcode
- Recovery Phrase ("Tap to reveal")
- Verify phrase ("Word #1")
- Final step (acknowledgements)
- Wallet ready ("Continue")

**Screenshots confirm:** All step titles, keypads, buttons, and text blocks are stacked vertically on a single screen.

**Cause hypothesis:** Every wizard step uses `style=move || format!("display:{};", ...)` to toggle between `"flex"` and `"none"`. Because closure-based inline styles are not applied on Android WebView, **no step ever receives `display:none`**. All `<div>` elements default to `display:block` and remain visible.

**Code location:** `app/src/src/pages/wallet.rs:268-696` — six `<div>` containers, each with reactive `style` closure controlling visibility.

**Status:** 🔴 CRITICAL — wallet creation flow is completely unusable.

**Problem:** Both "Create passcode" and "Confirm passcode" keypads are visible simultaneously, stacked one above the other.  
**Same root cause as #3:** The SetPin and ConfirmPin steps are hidden via reactive inline `display:none`, which is ignored by the WebView.

## Root Cause Analysis

### The pattern: static vs. reactive inline styles

| Pattern | Android WebView | Example | Status |
|---------|----------------|---------|--------|
| Static CSS class (`.keypad-grid`) | ✅ Applied | `passcode.rs` | Works |
| Static inline style (`style=format!("...")`) | ✅ Applied | `unlock.rs:209` | Works |
| Reactive inline style (`style=move \|\| format!("...")`) | ❌ Ignored | `wallet.rs:269`, `button.rs:71` | Broken |
| Reactive class (`class=move \|\| if ...`) | ⚠️ Unverified | `passcode.rs:34` | Needs test |

### Why this happens (most likely)

Leptos 0.7 CSR sets reactive attributes via JavaScript DOM APIs (`element.style.cssText` or individual `style.setProperty` calls) inside a microtask or `requestAnimationFrame` callback. On Android WebView initialized by Tauri/Wry, there may be a **timing mismatch** or **missing JS interop support** for reactive style mutations during the initial mount phase. The WASM module loads and executes, but the reactive effect queue that should flush initial styles does not write to the DOM.

**Evidence supporting this theory:**
1. `unlock.rs` renders correctly — it uses static `style=format!(...)` strings, which Leptos sets synchronously via `setAttribute` during initial mount.
2. `wallet.rs` and `button.rs` break — they rely on `Signal`-driven closures that Leptos subscribes to and updates reactively.
3. CSS classes from `main.css` apply fine — they are parsed by the browser independently of Leptos reactivity.

### Alternative hypotheses (to rule out)

| Hypothesis | Evidence for | Evidence against | Test |
|------------|-------------|------------------|------|
| CSP blocks inline styles | CSP has `style-src 'self' 'unsafe-inline'` | Static inline styles work; `'unsafe-inline'` is present | Check logcat for CSP violations |
| SRI on CSS link fails | Trunk adds `integrity` to `<link rel="stylesheet">` | `.keypad-grid` class works, so CSS loads | Remove `integrity` attr, rebuild, retest |
| `display:none` ignored by WebView | Old Android bugs | Chrome 123+ should support this perfectly | Test static `display:none` in raw HTML |
| `env(safe-area-inset-*)` breaks CSS | Used in `calc()` expressions | Unlock page uses same calc and works | N/A |

## Recommended Fix Strategy

### Option A: Replace reactive `style` with reactive `class` + CSS (Recommended)

Instead of toggling `display:none` via inline style closures, use CSS classes:

```css
/* Add to main.css */
.wizard-step {
    flex-direction: column;
    flex: 1;
    display: none;
}
.wizard-step.active {
    display: flex;
}
```

Then in `wallet.rs`:
```rust
<div class=move || if step.get() == Step::SetPin { "wizard-step active" } else { "wizard-step" }>
```

**Why this may work:** `class` reactive closures might use a different code path in Leptos (attribute vs. property), and even if they don't, `className` mutations are among the most robust DOM operations across all WebView versions.

### Option B: Use `<Show>` for conditional rendering

Replace `display:none` toggling with Leptos `<Show>` or `{move || if condition { view! { ... } } else { () }}`:

```rust
{move || if step.get() == Step::SetPin { view! {
    <div class="wizard-step">...</div>
}} else { () }}
```

**Risk:** If the root cause is broader (all reactive DOM operations fail), `<Show>` may also break. However, `app.rs` already uses `<Show for TabBar>` and it appears to function (tab bar shows/hides correctly on state change). This suggests `<Show>` might be safer than reactive `style`.

### Option C: Force synchronous style flush

If the issue is timing-related, adding a small `setTimeout` or `Effect` that manually sets styles after mount might work as a temporary band-aid. This is hacky and not recommended for production.

### Option D: Downgrade/fork Leptos attribute handling

If confirmed as a Leptos 0.7 bug on WebView, we could patch the framework's attribute-setting code to use `setAttribute('style', ...)` instead of `style.cssText` for the initial mount. This is a last resort.
